00-0

專案摘要

Executive Summary
01 · Problem
問題定義

一般民眾在尋找律師時，常面臨資訊分散、案件經驗難以判斷、公開風險資訊難比對等問題。

02 · Solution
解決方案

本系統整合法院判決、律師公開資訊、事務所資訊、新聞聲量與會員心得，提供複合條件搜尋、律師資料頁、風險資訊觀測與付費解鎖功能。

03 · Scope
MVP 範圍

第一階段聚焦民事案件與近五年公開資料，不設計勝率榜、戰力分數或主觀能力排名，避免過度定性與法遵風險。

詞彙表 Glossary

Glossary
📚 使用方式

本章節用於統一名詞與英文建議命名，避免在討論法遵、資料來源、權限與資料模型時產生歧義。請以表格的英文建議命名作為 Domain / API 命名基準（可依既有程式碼再做微調）。

詞彙英文建議命名定義備註
律師案件圖譜Lawyer Case Map本平台名稱，定位為協助使用者查詢、比較律師與事務所的法律科技平台。非法律諮詢平台
律師Lawyer平台中被查詢與比較的主要對象。來源應以公開資訊與判決資料為主
事務所LawFirm律師所屬組織，可作為背景與風評輔助資訊。一位律師可隸屬一個主要事務所
案件輪廓CaseProfile根據公開判決資料整理出的案件類型、法院、年份、經驗摘要等資訊。不等於勝率或能力評分
案件適配CaseFit系統根據案件類型、地區、經驗輪廓等資訊，輔助使用者篩選候選律師。不應宣稱保證適合
公開風險資訊RiskInfo來自公開來源的風險、爭議、負面聲量等整理資訊。應中性呈現
誠信風險觀測Integrity Risk Observation對公開爭議與可信來源的觀測性描述。避免使用「案底」等定性字眼
可信度分層CredibilityLevel對資料來源可信度進行分級，例如法院判決、官方資料、新聞、社群、會員心得。用於 UI 標示與權重設計
會員心得Review會員針對律師接觸、諮詢、委任流程等經驗提交的內容。需審核、檢舉與下架機制
付費解鎖PaidUnlock使用者透過付款取得完整資訊或會員心得的存取權。MVP 可先做單筆解鎖
已解鎖會員UnlockedMember已完成特定內容解鎖的會員。比「付費會員」更精準
申訴Dispute / Complaint律師或事務所針對平台內容提出更正、隱藏或下架請求。法遵重要流程
檢舉Report使用者針對會員心得或不當內容提出檢舉。與申訴不同
隱藏Hidden內容暫時不公開，但保留於後台。可恢復
下架 / 移除Removed內容不再公開，通常作為終局處理。建議保留審計紀錄
更正Corrected經查證後修正內容。不應直接覆蓋歷史紀錄
綠界ECPay第三方金流服務。處理付款與回呼

UML 圖表功能說明

UML Diagram Guide
UML 圖表本文件用途適合閱讀者回答的問題
Use Case Diagram描述系統有哪些角色，以及各角色可以使用哪些功能。PM、SA、工程師、設計師誰會使用系統？可以做什麼？
Activity Diagram描述一個業務流程從開始到結束的步驟與判斷。PM、SA、後端、前端使用者或系統流程怎麼走？有哪些分支？
Sequence Diagram描述前端、後端、資料庫、第三方服務之間的呼叫順序。後端、前端、架構師API 呼叫順序是什麼？誰呼叫誰？
Class Diagram描述核心資料物件、欄位與關聯。後端、DBA、架構師系統有哪些核心資料表或物件？彼此怎麼關聯？
State Machine Diagram描述某個物件的狀態如何轉換。後端、QA、PM訂單、心得、風險資訊有哪些狀態？什麼事件會改變狀態？
Component Diagram描述系統模組與外部服務的依賴關係。架構師、DevOps、後端系統應該拆成哪些模組？資料怎麼流動？
🧭 補充說明

本文件中的 UML 圖表不是最終實作限制，而是 MVP 階段的系統分析基準。若後續需求、法遵策略、商業模式或資料來源發生變更，應同步更新對應 UML 圖表與 Domain Model。

00-5

使用者情境

User Scenario
本頁重點

本情境用一位一般使用者尋找民事糾紛律師的過程，串接後續 Use Case、Activity、Sequence 與 Class Diagram，讓聽眾先理解系統實際使用方式。

01
輸入查詢條件

使用者輸入案件類型、地區、律師姓名或關鍵字，例如民事糾紛、台北地區、特定事務所等。

02
查看候選律師

系統根據律師基本資料、案件輪廓、事務所資訊與公開風險摘要，回傳候選律師名單。

03
查看免費資料頁

使用者進入律師資料頁，查看免費摘要、案件輪廓、事務所資訊與基礎風險提示。

04
登入並付費解鎖

若使用者想查看完整公開風險整理與會員心得，需登入並透過綠界完成付費解鎖。

05
查看完整內容

付款成功後，系統建立 UnlockRecord，使用者可查看完整案件資訊、公開風險整理與會員心得。

06
檢舉或申訴

若會員心得或公開風險資訊有爭議，可透過檢舉或申訴流程進入管理員審核。

00-6

MVP 系統邊界

System Boundary
本頁重點

系統邊界用來避免需求失焦，也能降低法律與名譽風險。本 MVP 定位為公開資訊查詢與比較輔助平台，不是法律諮詢平台，也不提供律師能力排名。

類別本 MVP 設計
系統定位律師公開資訊查詢與比較輔助平台
核心範圍民事案件、近五年公開資料、搜尋型平台
主要功能複合條件搜尋、律師資料頁、事務所資訊、公開風險資訊、會員心得、付費解鎖
不提供法律諮詢系統不回答具體法律問題，也不替使用者判斷訴訟策略
不保證律師適合度系統只提供資料整理與條件篩選，最終選擇仍由使用者自行判斷
不設計勝率榜不使用「勝率排行」、「最強律師」、「戰力分數」等過度定性功能
不直接撮合委任MVP 不處理律師委任契約、收費議價或案件媒合流程
不將會員心得作為事實結論會員心得屬主觀經驗，需審核與檢舉機制，僅作為參考資訊
🧭 邊界說明

本 MVP 的核心價值是「整理公開資料並協助比較」，而不是判斷律師優劣。透過明確系統邊界，可避免功能設計過度承諾，也能讓 UML 圖表中的角色、流程與資料模型維持在可落地範圍。

00-7

資料來源可信度分層

Data Credibility Levels
本頁重點

本系統資料來源包含法院判決、官方公開資訊、新聞、社群與會員心得。不同來源可信度不同，因此需要透過 CredibilityLevel 標記，避免所有資料被視為同等可信。

等級資料來源說明系統使用方式
Level 1法院判決、官方律師公開資訊可信度最高，作為案件輪廓與基本資料主要依據可進入 Lawyer、LawFirm、CaseProfile 基礎資料
Level 2新聞媒體、公開報導需標示來源、發布時間與摘要，不直接作為定性結論可進入 RiskInfo，但需人工審核與狀態控管
Level 3社群討論、會員心得主觀性高，容易受個人經驗影響僅作為參考，不作為系統評分依據
Unknown尚未確認來源或格式異常資料資料仍需驗證或清洗不應直接公開，需進入 PendingVerification 或 Failed 匯入紀錄
📊 對應資料模型

此分層對應 Class Diagram 中的 DataSource.credibilityLevel 與 RiskInfo.credibilityLevel。資料匯入流程會在建立或更新 CaseProfile / RiskInfo 時，同步標記資料來源與可信度等級。

00-8

法遵與名譽風險控制

Compliance & Reputation Risk Control
本頁重點

此題目涉及律師、公開爭議資訊與會員心得，因此系統不能只設計搜尋與付款，還必須設計審核、申訴、更正、隱藏、下架與資料可信度標示。

風險對應設計說明
不實評論Review 審核流程會員心得需經系統規則檢查與管理員審核才可上架
誹謗或過度定性中性命名與 RiskInfo 狀態機使用「公開風險資訊」、「誠信風險觀測」等中性名稱，避免使用「案底」等字眼
資料來源不可信DataSource / CredibilityLevel法院判決、新聞、社群與會員心得需區分可信度
當事人異議Complaint / Dispute 流程律師或事務所可提出更正、隱藏或下架請求
內容爭議處理Hidden / Corrected / RemovedHidden 為暫時隱藏，Corrected 為更正，Removed 為終局下架
付款爭議PaymentOrder 狀態機記錄付款、解鎖、失敗、逾時與退款等狀態
黑盒推薦風險可解釋的搜尋與排序依據系統以案件類型、地區、資料來源與風險摘要作為輔助依據，不產生不可解釋的律師能力分數
⚖️ 報告說明

本系統的風險控制不是單一功能，而是分散在 Review 狀態機、RiskInfo 狀態機、Complaint 類別、DataSource 可信度分層與 Admin Module 中。這些設計共同形成平台的內容治理機制。

Paid Unlock Sequence
📐 設計說明

【需求推導】 付費解鎖使用綠界（ECPay）金流（需求 §7.5）。解鎖後建立 UnlockRecord，供後續查詢是否已解鎖。

【合理補充】 Backend 檢查「已解鎖」邏輯（避免重複付款）為標準金流實作慣例。ECPay webhook 驗簽步驟為綠界標準串接規格。

報告重點

此圖展示前端、後端、Payment Service、ECPay 與資料庫之間的呼叫順序，並特別處理未登入、已解鎖與付款驗證失敗情境。
