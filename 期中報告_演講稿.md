## 開場（1 分鐘）

> 各位老師、各位同學大家好。
>
> 我們這次期中報告的題目是「**律師案件圖譜**」 — 一個協助一般使用者查詢、比較律師的法律科技平台。我們的目標不是做出真正能上線的產品，而是把這個題目當作載體，**把這學期目前學到的需求分析、Use Case、Class Diagram 等 UML 工具實際走過一次**。
>
> 接下來我會用大約 20 分鐘的時間，按照系統分析的思考脈絡，從問題定義、需求、UML 圖表到法遵設計，帶大家走一輪我們的 MVP 系統規劃。

**過場**：先讓大家看一下整份報告的導覽。

## §00 報告導覽（45 秒）

**演講重點**
- 報告分五段：(1) 背景與問題 (2) MVP 範圍 (3) UML 建模 (4) 法遵設計 (5) 系統架構與擴充
- 重點放在「為什麼這樣設計」，不是「圖怎麼畫」

**順稿**
> 整份報告我們分成五個段落：先說明為什麼要做這個系統，再聚焦在 MVP 範圍，接著用 UML 圖表呈現系統設計，特別會強調風險與法遵的設計，最後說明整體架構與後續擴充方向。

**過場**：先給大家一個專案摘要的全貌。

## §00-0 專案摘要（45 秒）

**演講重點**
- **Problem**：找律師很難，資訊分散、案件經驗難判斷
- **Solution**：整合判決、律師公開資訊、新聞、事務所與會員心得，提供複合搜尋與付費解鎖
- **Scope**：MVP 限定民事案件、近五年，**不做**勝率排名等過度定性功能

**順稿**
> 簡單講，我們要解決的問題是：一般民眾找律師時，資訊分散、案件經驗難以判斷、又難確認誠信背景。我們的解決方案是整合公開判決、律師公開資訊、新聞、事務所背景與會員心得，讓使用者能用複合條件搜尋律師。MVP 階段我們特別限定在民事案件、近五年資料，並且**刻意不做**勝率榜、戰力分數這類過度定性的功能 — 因為這些容易引發名譽爭議，是我們在 §00-8 會專門講的法遵考量。

**過場**：等一下會用一張整體設計概覽帶大家看圖表清單。

## §00-1 設計概覽（30 秒）

**演講重點**
- 全篇 10 張 UML 圖、18 份 Use Case Description、6 章需求 / 法遵 / 架構說明
- 每張圖都標 **需求推導** / **合理補充**，避免老師問「這從哪來的？」

**順稿**
> 為了讓每個設計決策可追溯，我們把每張圖的內容區分為兩類：綠色標籤代表來自需求訪談文件，橘色標籤代表為了 UML 完整性所做的合理補充。這樣評分時可以清楚看到哪些是依據、哪些是延伸。

**過場**：在進入圖之前，先統一一下用語。

## §00-2 詞彙表 Glossary（30 秒，可快速帶過）

**演講重點**
- 重點不是逐項唸，而是強調我們做了「術語中性化」處理
- 例如：不用「案底」 → 改用「**公開風險資訊**」、「**誠信風險觀測**」

**順稿**
> 詞彙表的設計重點在於**用詞中性化**，避免使用「案底」這類有罪推定的字眼，改用「公開風險資訊」、「誠信風險觀測」這些觀測性的描述。這也對應到法遵風險控制的設計原則。

**過場**：UML 圖表功能說明我跳過細節，每張圖用途其實在後面會直接看到。

## §00-3 UML 圖表功能說明 / §00-4 需求追蹤矩陣（合併講，45 秒）

**演講重點**
- UML 指南是給聽眾的速查表，不細講
- **需求追蹤矩陣**重點：每筆需求都對應到具體的 UML 圖和類別 / 模組

**順稿**
> 需求追蹤矩陣這頁很重要 — 它確保我們**每一條需求都被至少一張圖覆蓋**，沒有遺漏。也方便老師抽查任意需求，都能找到對應的設計。

**過場**：接下來透過使用者情境讓大家「感受」這個系統怎麼用。

## §00-5 使用者情境（1 分鐘）

**演講重點**
- 6 步：輸入條件 → 看候選 → 看免費頁 → 登入解鎖 → 看完整內容 → 檢舉/申訴
- **這是後面所有 UC 的故事主線**

**順稿**
> 想像一位使用者有民事糾紛，想找台北地區、有類似案件經驗的律師。他輸入條件後系統回傳候選名單；他點進其中一位律師看免費資料；如果想看完整風險整理跟會員心得，要登入並透過綠界付費解鎖；解鎖後可以看完整內容；如果發現有不當資訊或心得，可以檢舉或申訴。**這 6 步驟就是我們後面所有 UC、Activity、Sequence 圖的故事骨架**。

**過場**：但這個系統不能無限擴大，所以我們明確劃出系統邊界。

---

## §00-6 MVP 系統邊界（45 秒）

**演講重點**
- 是**查詢與比較輔助平台**，**不是**法律諮詢、不保證適合度、不做勝率榜、不撮合委任
- 把「不做什麼」明確寫出來是法遵與聲譽保護的設計

**順稿**
> 我們刻意把系統定位寫得很清楚：這是一個**查詢與比較輔助平台**，**不是**法律諮詢。我們**不**回答具體法律問題、**不**保證律師適合度、**不**做勝率排行、**不**撮合委任。這些「不做」的部分跟「做」一樣重要 — 它們保護我們不會在 MVP 階段就踩到法遵紅線。

**過場**：而那些我們會做的資訊整合，可信度其實不一樣，所以我們分了三層。

---

## §00-7 資料來源可信度分層（45 秒）

**演講重點**
- Level 1：法院判決、官方資料（最高）
- Level 2：新聞媒體
- Level 3：社群、會員心得（最低，僅參考）
- Unknown：未驗證，不直接公開

**順稿**
> 我們把資料來源分四層：法院判決跟官方資料是 Level 1，最可信；新聞媒體 Level 2，需要審核；社群跟會員心得 Level 3，**僅作參考、不能作為評分依據**；未驗證的資料則暫不公開。每筆資料在系統內都會帶 CredibilityLevel 欄位，UI 也會顯示來源等級，讓使用者自己判斷。

**過場**：這個分層直接接到我們最重要的設計 — 法遵與名譽風險控制。

---

## §00-8 法遵與名譽風險控制（1 分鐘 — 強調這頁）

**演講重點**
- 七種風險、七種設計對策
- 強調我們**不是設計搜尋與付費就好**，而是把審核、申訴、更正、隱藏、下架整套機制都建進系統
- 這也是我們最花心思的一塊

**順稿**
> 這個題目最敏感的地方是律師、爭議資訊、會員心得 — 都很容易引發名譽糾紛。所以我們不只設計搜尋與付款，還把整套**內容治理**機制建進系統：
>
> - 不實評論 → Review 審核流程
> - 誹謗風險 → 中性命名 + RiskInfo 狀態機
> - 資料不可信 → CredibilityLevel 分層
> - 當事人異議 → Complaint / Dispute 流程
> - 內容爭議 → Hidden / Corrected / Removed 三層處理
> - 黑盒推薦風險 → 推薦依據都可解釋，**不**產生黑盒分數
>
> 這些設計散佈在後面好幾張 UML 圖裡，我們等下看到 Review 狀態機跟 RiskInfo 狀態機時會回扣這頁。

**過場**：講完背景與設計原則，接下來進入正式的 UML 建模。先看需求清單。

## §00-9 需求清單（1 分鐘）

**演講重點**
- 16 條需求，前綴依領域分：S 搜尋 / M 會員 / P 付費 / R 心得 / A 管理 / D 資料匯入
- 每條需求用「**作為 XX，我希望 YY，以便 ZZ**」格式（user story）
- 對應 UC 編號

**順稿**
> 我們把訪談文件整理成 16 條編號需求，前綴依領域分類：搜尋類 REQ-S、會員類 REQ-M、付費類 REQ-P，依此類推。每一條都用 user story 格式撰寫 — 作為什麼角色、希望做什麼、為什麼。最右邊欄位標明對應到哪個 Use Case，後面 UC Description 也會反向回引這些 REQ 編號。

**過場**：有了需求，下一步是把它們拆成事件。

## §00-A 事件表（1 分鐘）

**演講重點**
- 14 筆事件，依教材格式：事件名稱 / 觸發點 / 來源 / 活動 / 回應 / 目的地
- 強調 **Timer 來源**（排程觸發）— 對應教材 Use Case 章節的 Timer Actor
- 事件表 → 推導出 Use Case

**順稿**
> 在 Use Case 前，我們先做事件表 — 這是教材中明確建議的需求分析工具。從系統外部觀點，每一行記錄「什麼事件觸發系統執行什麼活動」。注意有幾筆事件的來源是 **Timer**，這就是教材提到的 Timer Actor — 由系統排程觸發的事件。事件表是我們推導 UC 的依據，每筆事件對應一個或多個 UC。

**過場**：好，接下來進入正式的 Use Case Diagram。

## §01 Use Case Diagram（2 分鐘 — 報告核心之一）

**演講重點**
- **7 個 Actor**：訪客 / 會員 / 已解鎖會員 / 管理員 / 律師代表 / 綠界 / 外部資料來源
- **Actor 一般化**：Member ⊳ Guest、UnlockedMember ⊳ Member（解釋這就是 OO 課堂講的繼承）
- **include vs extend**：
  - UC7 發起付費解鎖 → include UC17 建立付款訂單、UC18 回呼付款結果（**內含子流程**）
  - UC8 查看付費內容 → extend UC3 查看免費資料頁（**已解鎖條件下的擴充**）
- 18 個 UC 分 6 組：搜尋瀏覽 / 會員管理 / 會員心得 / 管理功能 / 資料匯入 / 金流處理

**順稿**
> 系統一共 7 個角色、18 個 Use Case。請注意三個 Actor 之間的一般化關係：會員繼承訪客的所有功能、已解鎖會員繼承會員 — 這跟物件導向的繼承是同一個概念，可以避免在每一個共用功能上重複連線。
>
> 兩個重要的 UC 關係：UC7「發起付費解鎖」用虛線箭頭加 include 標籤指向 UC17 跟 UC18，代表「發起解鎖」必定包含「建立付款訂單」跟「處理付款回呼」這兩個子流程。而 UC8「查看付費內容」extend UC3「查看免費資料頁」— 意思是當使用者已解鎖時，免費頁上會擴充顯示付費內容。
>
> 一個重要的設計決定：我們把 ECPay 綠界當作 **System Actor**（系統角色），這對應教材中提到 Actor 不一定是真人，也可以是外部系統。

**可能被問**
- Q: 為什麼把已解鎖會員獨立成一個 Actor 而不是用會員的屬性？
  - A: 因為它對應不同的權限邊界 — 教材也提到「Actor 是角色，不是特定的人」，這比在類別裡判 if 更清楚，且未來訂閱制時可擴充。
- Q: include 跟 extend 為什麼這樣分？
  - A: include 是「一定會包含」，extend 是「特定條件下才擴充」。付款一定要建訂單、處理回呼，所以是 include；查看免費頁時是否擴充付費內容，要看有沒有解鎖紀錄，所以是 extend。

**過場**：使用案例圖只表達高層級的「系統做什麼」，每個 UC 後面都還要寫詳細描述。

## §01-A Use Case Description（3 分鐘）

**演講重點**
- **18 份描述**，每份依教材範本欄位：相關需求 / 完成目標 / 前置條件 / 成功失敗結束狀態 / 主要次要角色 / 觸發事件 / 主要流程 / 延伸流程
- 不要 18 份都唸，挑 **3 份** 示範：
  1. **UC01 搜尋律師**（最基本、流程清晰）
  2. **UC07 發起付費解鎖**（最有 include、含金流）
  3. **UC17 建立付款訂單**（**被 include 的子 UC** — 主要角色填「無」，符合教材慣例）

**順稿**
> 18 份描述全部依教材範本撰寫。我挑三份來示範。
>
> **UC01 搜尋律師** — 主要角色是訪客或會員，主要流程 6 步：輸入條件、送出、系統驗證、並行查詢、排序、回傳。延伸流程處理條件為空或查詢例外。這是最典型的查詢類 UC。
>
> **UC07 發起付費解鎖** — 步驟 3 跟步驟 6 我們用 `Include::Create Payment Order` 跟 `Include::Handle Payment Callback` 標示，代表呼叫子 UC。延伸流程處理未登入、已解鎖、付款逾時、驗簽失敗四種情境。
>
> **UC17 建立付款訂單** — 注意它的「主要角色」跟「次要角色」都填「無」，因為它**不是被 Actor 直接觸發**，而是被 UC07 include 進來。觸發事件寫成「解鎖請求由 UC07 送達」。這完全對應教材範本第 5 頁 Find Parts 的寫法。

**可能被問**
- Q: 為什麼有些 UC 主要角色填「無」？
  - A: 教材範本明確示範 — 被 include 的子 UC 沒有 Actor 直接觸發，所以主要角色填「無」、觸發事件用「資料 / 要求被送達」描述。

**過場**：靜態 UC 講完，接下來看流程。

## §03 Sequence Diagram：付費解鎖流程（1.5 分鐘 — 強調這張）

**演講重點**
- 5 個參與者：使用者 / 前端 / 後端 / Payment Service / ECPay / Database
- 三個 alt 分支：**未登入 / 已解鎖 / 未解鎖**
- 已解鎖直接短路回傳，**不重複建單**
- ECPay 回呼後的 **三步狀態轉移**：Paid → 建立 UnlockRecord → UnlockGranted
- **驗簽失敗時不更新訂單也不建解鎖**（重要安全考量）

**順稿**
> 這張圖比較大，重點看三件事：
>
> **第一**，三個 alt 分支處理未登入、已解鎖、未解鎖三種狀態。已解鎖的情況下直接短路回傳完整內容，**不會重複建單** — 這對應 UC07 延伸流程 2.2。
>
> **第二**，ECPay 回呼後的處理順序：先驗簽，驗簽過了才依序更新 PaymentOrder 為 Paid、建立 UnlockRecord、最後 UnlockGranted。這三步驟對應 §07 PaymentOrder 狀態機的狀態轉移。
>
> **第三**，**驗簽失敗的處理**：我們特別寫了 `BE-->ECPay: 回應 FAIL`，且**不更新訂單、不建立解鎖紀錄**。這是金流安全的標準做法 — 即使收到 webhook，沒驗簽過就什麼都不做。

**可能被問**
- Q: idempotencyKey 為什麼放在 PaymentOrder？
  - A: 防止重複建單。如果使用者點兩次「解鎖」，我們用 idempotencyKey 確保只有一筆訂單。

**過場**：金流講完，再看一個跟法遵直接相關的流程 — 心得審核。

## §04 Activity Diagram：會員心得投稿與審核（1 分鐘）

**演講重點**
- 心得**不是直接上架**
- 兩道關卡：(1) 系統規則檢查（字數、禁用詞、重複投稿）(2) 管理員審核
- 管理員三種決定：通過 / 要求修改 / 拒絕
- 「要求修改」會循環回去，對應 Review 狀態機的 NeedRevision → PendingReview

**順稿**
> 會員心得有兩道關卡：系統自動檢查（字數、禁用詞、重複投稿），通過後狀態變成 PendingReview，再由管理員人工審核。管理員三種決定：通過、要求修改、拒絕。要求修改的話會走 NeedRevision 狀態，會員修改後再送審 — 這個循環在 §06 狀態機會看得更清楚。
>
> **為什麼這樣設計？** 因為心得直接影響律師名譽，如果讓使用者自由發布，平台會承擔誹謗風險。所以審核機制是法遵設計的核心之一。

**過場**：流程講完了，現在看資料模型。

## §05 Class Diagram（3 分鐘 — 報告核心之二）

**演講重點**
- **10 個核心類別**，全部從需求文件推導
- **多重性都標示**（1, 0..\*, 1..\* 等）
- **三種關係強度**：
  - 聚合（空心菱形 `o--`）：LawFirm 跟 Lawyer — 事務所解散律師仍存在
  - 組合（實心菱形 `*--`）：Lawyer 跟 CaseProfile — 律師沒了案件輪廓也沒意義
  - 一般關聯（實線）：User 跟 Review、PaymentOrder、UnlockRecord
- **敘述性操作**：每個類別都有 `+register()`、`+approve()` 這類操作 — 對應教材「分析階段只寫敘述性操作」
- **多型 target 設計**：Complaint 跟 RiskInfo 可以掛在 Lawyer 或 LawFirm 上

**順稿**
> 10 個類別、四種關係。重點看幾個設計決定：
>
> **第一**，**多重性都標了** — 例如 LawFirm `1` 對 Lawyer `0..*`，因為一間事務所可以沒有律師（剛成立）也可以有很多律師。這個習慣是教材強調的。
>
> **第二**，**聚合與組合的差別**：LawFirm 跟 Lawyer 是聚合（空心菱形）— 事務所解散後律師還在；但 Lawyer 跟 CaseProfile 是組合（實心菱形）— 律師資料如果不存在，案件輪廓也沒意義。這對應教材 §關聯關係 16~18。
>
> **第三**，每個類別都有**敘述性操作**：User 有 `+register()`、`+submitReview()`；Review 有 `+approve()`、`+requestRevision()`。教材明確說分析階段**只寫敘述性操作，不寫實作細節**，所以我們不會有 `+register(email: string): User` 這種型別細節。
>
> **第四**，Complaint 跟 RiskInfo 採**多型 target 設計** — targetType + targetId 可以指向 Review / RiskInfo / Lawyer / LawFirm 任何一種目標。這在 MVP 階段比較彈性，但資料一致性需要應用層補強。

**可能被問**
- Q: 為什麼 PaymentOrder 跟 UnlockRecord 分兩個類別？
  - A: 訂單可能失敗、退款 — 跟「成功解鎖」的概念不同。一張訂單成功才產生一筆解鎖。
- Q: 為什麼 User 用 enum role 而不是繼承？
  - A: MVP 階段 role 變動少，用 enum 簡單。但在 Use Case Diagram 我們用了 Actor 一般化來表達角色關係。

**過場**：類別資料結構講完，但資料本身有狀態 — 接下來三張狀態機。

## §06 State Machine：Review 狀態機（45 秒）

**演講重點**
- 7 個狀態：Draft → PendingReview → Published / Rejected / NeedRevision → Reported → Hidden → Removed
- 三條關鍵路徑：
  - **正常**：Draft → PendingReview → Published
  - **檢舉**：Published → Reported → Hidden / Removed
  - **修改循環**：PendingReview → NeedRevision → PendingReview

**順稿**
> Review 狀態機的重點是它表達了我們的審核設計：草稿 → 待審 → 上架 → 被檢舉 → 隱藏 / 移除。Rejected 跟 Removed 都是終局狀態（指向 \[\*]）。Hidden 是**可恢復**的暫時狀態，Removed 是**不可逆**。

**過場**：付款訂單也有類似的狀態演進。

---

## §07 State Machine：PaymentOrder 狀態機（45 秒）

**演講重點**
- 主要路徑：Created → PendingPayment → Paid → UnlockGranted
- 三條失敗路徑：Failed / Expired / RefundRequested → Refunded
- 對應 §03 Sequence 圖步驟

**順稿**
> 付款狀態機跟 Sequence 圖一一對應。主路徑四步：Created（建單）、PendingPayment（等綠界）、Paid（綠界回呼成功）、UnlockGranted（建立解鎖）。失敗有三條路徑：付款失敗、逾時、退款。退款設計是消保法鑑賞期的合理預設。

**過場**：第三個狀態機跟法遵直接相關。

---

## §08 State Machine：RiskInfo 狀態機（45 秒 — 強調這張）

**演講重點**
- 8 個狀態，特別有 **Disputed**（律師申訴中）跟 **Corrected**（修正後）
- Corrected 狀態**不直接覆蓋原始紀錄**，保留審計軌跡
- 對應 §00-8 法遵設計，§01 UC13 處理申訴

**順稿**
> 這張是法遵設計最直接的體現。風險資訊不能直接公開，要走 Collected → PendingVerification → Verified → Published；上架後如果律師提出申訴會進 **Disputed**，管理員確認後可能進入 Corrected、Hidden 或維持 Published。
>
> **Corrected 狀態的設計很關鍵** — 它不是覆蓋原始資料，而是建立一個修正版本，**保留審計軌跡**，這是法遵與資料治理的標準做法。

**過場**：所有資料模型跟狀態講完，最後看系統架構。

## §09 Component Diagram（1 分鐘）

**演講重點**
- 14 個元件，前端 / 後端 / 資料層 / 外部系統四層
- 後端拆 8 個模組對應 §7 功能需求
- **Search Index 獨立**（MeiliSearch / Elasticsearch）— 為了支援複合條件查詢
- **Auth Module 獨立**（JWT / OAuth2）— 安全設計慣例
- **Data Ingestion 是後台排程服務**

**順稿**
> 系統拆四層：前端有 Public Web App（給使用者）跟 Admin Web Console（給管理員）；後端是 REST API 加 8 個核心模組；資料層分 PostgreSQL 跟 Search Index；外部串綠界跟資料來源。
>
> 重點是 Search Index 獨立 — 因為複合條件查詢直接打資料庫效能會差，所以用 MeiliSearch 之類的搜尋引擎。Data Ingestion 是排程服務，定期匯入新資料並更新索引。

**過場**：最後看資料匯入流程。

## §10 Activity Diagram：資料匯入流程（45 秒）

**演講重點**
- 入口三種：法院判決 / 律師公開資訊 / 新聞社群
- 共同步驟：解析 → 去重 → 建/更新類別 → 標記 CredibilityLevel → 更新索引
- 失敗時記錄 Failed 批次，不污染主資料

**順稿**
> 資料匯入流程的關鍵是**所有匯入資料都會被標記 CredibilityLevel**：法院判決 Level 1、新聞 Level 2、社群 Level 3。如果格式驗證失敗，會記錄 Failed 批次，不會把髒資料寫進主資料庫。新聞跟社群資料不會直接上架，而是進 PendingVerification 等管理員審核。

## Q&A 預想題庫

### 評分相關
- **Q1：為什麼 UC 圖中 Actor 跟 UC 連線沒箭頭？**
  - 教材明確規定 Actor 與 UC 連線沒有方向性。我們的 Mermaid 圖用 `---` 表示無方向實線。
- **Q2：為什麼有些 UC Description 主要角色填「無」？**
  - 被 include 的子 UC（如 UC17、UC18）不是 Actor 直接觸發，依教材範本主要角色應填「無」、觸發事件描述為「資料 / 要求被送達」。
- **Q3：你們用 Mermaid 不用 PlantUML，會不會影響評分？**
  - 我們在每張圖下方都附 PlantUML 原始碼附錄，內容一致，只是 Mermaid 在 HTML 中可即時渲染、適合上台報告。

### 設計決策
- **Q4：為什麼不一開始就做完整功能？**
  - MVP 範圍鎖定民事 + 近五年，是為了快速驗證核心搜尋與付費機制，再依使用者反饋擴展。
- **Q5：付費內容跟免費內容怎麼劃分？**
  - 免費：基本背景、案件輪廓摘要、基礎風險提示。付費：完整案件資訊、完整風險整理、會員心得。對應 §7.2 需求。
- **Q6：如何防止刷單或刷心得？**
  - 付款用 idempotencyKey 防重複建單；心得用系統規則檢查 + 管理員審核兩道關卡。

### 法遵
- **Q7：律師提出申訴後系統怎麼處理？**
  - 走 UC13「處理申訴 / 更正」流程：建立 Complaint、目標狀態變 Disputed、管理員審視、決定維持/修正/隱藏/下架、通知申訴方。對應 §08 RiskInfo 狀態機。
- **Q8：會員心得可信度怎麼處理？**
  - 心得屬 Level 3，僅作參考，不作為系統評分依據。系統 UI 會明確標示「會員主觀經驗」。

### 技術
- **Q9：為什麼搜尋要獨立 Search Index？**
  - 複合條件查詢（姓名、案件類型、地區、爭議、聲量、經驗）直接打 RDB 效能差，用 MeiliSearch / Elasticsearch 可大幅提升搜尋速度。
- **Q10：綠界回呼為什麼要驗簽？**
  - 防止偽造 webhook。任何人都可以發 POST 到我們的 callback 端點，沒驗簽就更新訂單會造成詐欺風險。
